\# UX Journey to Slice Map



## 目的



本文件定義使用者旅程（UX Journey）與系統能力切片（Workflow Slice）之間的對應關係。



它回答：



- 哪些使用者旅程需要哪些 slice 就位才能通過

- 每個旅程的驗證點是什麼（對話回覆 + UI 雙面驗證）

- 哪些旅程是 founder human gate 的必測情境

- 旅程之間的依賴順序



它不回答：



- slice 的內部實作細節（見各 L-spec）

- benchmark oracle 設計（見 L5B）

- 自動化測試設計（見 L5A）



---



## 關係說明：本文件 vs L0 Product Capability Spec



`L0\\\_PRODUCT\\\_CAPABILITY\\\_SPEC.md` 定義的是**產品能力的靜態清單**：系統能做什麼、每個能力域的邊界是什麼。



本文件定義的是**使用者實際走過的動態旅程**：使用者在什麼情境下觸發什麼能力、系統應如何回應、驗證點在哪裡。



兩者互補，不重疊：



- L0 是能力地圖

- 本文件是旅程劇本與驗證清單



本文件應在 L0 能力就位後建立，並隨 slice 推進持續補充。



---



## 驗證原則



**所有旅程的驗證必須同時包含兩個面向：**



1\. **對話回覆驗證**：系統在 chat 中的即時回覆是否正確（數字、處置、說明）

2\. **UI 狀態驗證**：Today 頁面 / body-plan 頁面的數字是否與對話一致



只驗證 UI 而不驗證對話回覆，不算通過。

只驗證對話回覆而不驗證 UI 同步，不算通過。

## 旅程 A：Onboarding 完整流程



### 情境



使用者第一次進入產品，完整填寫 onboarding 資訊。



### 前提



無（這是所有旅程的前置條件）



### 測試劇本



1\. 使用者開啟 UI onboarding 表單

2\. 填寫：性別、身高、體重、年齡、目標（減重）

3\. 填寫：活動量（例如久坐）、目標體重、每週目標速率（例如 0.5 kg/week）

4\. 系統顯示初始計畫 proposal，包含：

&#x20;  - 估計 TDEE（例如 1800 kcal）

&#x20;  - 每日目標熱量（例如 2000 kcal）

&#x20;  - 每週預計減重速率

5\. 使用者確認計畫

6\. 系統建立 active BodyPlan 與今日 DayBudgetLedger



### 驗證點



**對話回覆驗證：**

- 使用者在 chat 問「我今天的目標是多少？」

- 系統回覆應包含：TDEE 數字、每日目標熱量數字（例如「你的 TDEE 約 1800 kcal，每日目標 2000 kcal」）

- 不可只說「已設定」而不給數字



**UI 驗證：**

- Today 頁面顯示目標熱量（例如 2000 kcal）

- 已消耗 0 kcal

- 剩餘 2000 kcal

- body-plan 頁面顯示 TDEE、目標體重、每週速率



**一致性驗證：**

- chat 回覆的數字 = UI 顯示的數字



### 邊界情境



- 跳過 Step 2（活動量等建議填欄位）：系統採 sedentary 預設，仍可完成 onboarding

- `weekly\\\_target\\\_rate\\\_kg` 超過 1.0 kg/week：系統應拒絕並說明上限

## 旅程 B：單回合錄入 + 預算同步



### 情境



使用者完成 onboarding 後，記錄一餐，確認預算正確更新。



### 前提



旅程 A 已完成（active BodyPlan 存在）



### 測試劇本



1\. 使用者在 chat 說「我剛吃了一碗滷肉飯和一杯無糖豆漿」

2\. 系統估算熱量並回覆（例如「滷肉飯約 550 kcal，無糖豆漿約 80 kcal，共約 630 kcal，已記錄」）

3\. 使用者問「我今天還能吃多少？」

4\. 系統回覆剩餘熱量（例如「今天目標 2000 kcal，已吃 630 kcal，還剩約 820 kcal」）



### 驗證點



**對話回覆驗證：**

- 錄入回覆應包含：食物名稱、估算熱量、已記錄確認

- 剩餘預算回覆應包含：今日目標、已消耗、剩餘三個數字

- 數字應與 UI 一致



**UI 驗證：**

- Today 頁面：已消耗更新為 630 kcal（或系統估算值）

- 剩餘熱量正確更新

- 餐點列表出現剛記錄的餐點



**一致性驗證：**

- chat 回覆的剩餘熱量 = UI 顯示的剩餘熱量

## 旅程 C：珍珠奶茶錄入（需確認糖量與大小杯）



### 情境



使用者說「我喝了一杯珍珠奶茶」，系統可以先記一筆有不確定性的估算，並追問糖量與杯型來提升精度。



### 前提



旅程 A 已完成



### 測試劇本



1\. 使用者在 chat 說「我喝了一杯珍珠奶茶」

2\. 系統應給出估算範圍或粗估採用值，並追問：

&#x20;  - 糖量（全糖 / 少糖 / 半糖 / 微糖 / 無糖）

&#x20;  - 杯型（大杯 / 中杯 / 小杯）

3\. 使用者回答「半糖大杯」

4\. 系統用新資訊 refinement/correction 原估算（例如「半糖大杯珍珠奶茶約 450–550 kcal，已更新為約 500 kcal」）



### 驗證點



**對話回覆驗證：**

- 第一則回覆：系統應提供 logged estimate，並反問糖量和杯型

- 第二則回覆（使用者回答後）：系統應給出 refinement/correction 後的估算範圍 + 採用值 + 已記錄確認

- 不可說「無法估算」或「請提供更多資訊」後就停止



**UI 驗證：**

- 使用者回答後，Today 頁面應更新消耗熱量

- 餐點列表出現珍珠奶茶記錄



### 邊界情境



- 使用者說「我喝了一杯珍奶，全糖大杯」（一次給完）：系統應直接估算，不需再問

## 旅程 D：家常菜錄入（需反問具體內容）



### 情境



使用者說「我剛吃了我媽做的家常菜」，系統無法估算，應反問具體菜色。



### 前提



旅程 A 已完成



### 測試劇本



1\. 使用者在 chat 說「我剛吃了我媽做的家常菜」

2\. 系統**不應**說「無法估算」或「家常菜難以精確計算」後停止

3\. 系統應反問具體內容，例如：

&#x20;  - 「可以告訴我大概吃了什麼菜嗎？例如炒青菜、紅燒肉、蛋花湯之類的？」

4\. 使用者回答「炒空心菜、紅燒肉三塊、白飯一碗」

5\. 系統估算並記錄（例如「炒空心菜約 80 kcal、紅燒肉三塊約 300 kcal、白飯一碗約 280 kcal，共約 660 kcal，已記錄」）



### 驗證點



**對話回覆驗證：**

- 第一則回覆：系統應反問具體菜色，不應說「無法估算」後停止

- 第二則回覆：系統應逐項估算並給出總計，確認已記錄

- 估算應合理（不應給出明顯偏低或偏高的數字）



**UI 驗證：**

- 使用者回答後，Today 頁面應更新消耗熱量



---



## 旅程 E：超標後的 UI 警告與對話回覆



### 情境



使用者今日攝取超過每日目標，系統應在對話和 UI 都顯示超標警告。



### 前提



旅程 A 已完成，今日已有部分錄入



### 測試劇本



1\. 使用者今日目標 1450 kcal，已記錄 1200 kcal

2\. 使用者說「我剛吃了一個排骨便當」（約 800 kcal）

3\. 系統記錄後，總計 2000 kcal，超出目標 550 kcal



### 驗證點



**對話回覆驗證：**

- 錄入回覆應包含：

&#x20; - 排骨便當的估算熱量

&#x20; - 今日已超出目標的明確說明（例如「今天已超出目標約 550 kcal」）

&#x20; - 不可只說「已記錄」而不提超標

- 不應在同一則回覆裡附加 rescue 建議（rescue 是獨立訊息）



**UI 驗證：**

- Today 頁面應顯示超標警告（例如紅色提示、「已超出 550 kcal」）

- 剩餘熱量顯示為負數或明確的超標狀態

- 不可只顯示數字而不顯示警告狀態



**一致性驗證：**

- chat 回覆的超標量 = UI 顯示的超標量

## 旅程 F：救援計畫（當日超標，主動詢問）



### 情境



使用者今日超標後，主動在 chat 詢問如何補救。



### 前提



旅程 E 已發生（今日超標狀態存在）



### 測試劇本



1\. 使用者在 chat 說「我今天超標了，怎麼辦？」

2\. 系統計算超標量（例如 550 kcal）

3\. 系統提出救援方案（例如「建議分 3 天補回來，每天少吃約 183 kcal」）

4\. 系統提供 proposal primary actions：接受方案 / 先不要；若想調整強度，使用者可在 chat 說「緩一點」或「短一點」

5\. 使用者點擊「接受這個方案」

6\. 系統確認並說明接下來幾天的調整



### 驗證點



**對話回覆驗證：**

- 救援訊息應是獨立的一則訊息（不夾帶在錄入回覆裡）

- 應包含：超標量、建議天數、每日回收量

- 應提供 proposal primary actions，且不把所有調整選項攤平成同級按鈕

- 接受後應回覆確認，說明接下來幾天的每日目標



**UI 驗證：**

- 接受後，Today 頁面應反映 rescue overlay（例如明日目標調整）

- Proposal inbox 應顯示已接受的救援方案



**邊界情境：**

- 使用者說「這樣也太硬了吧」：系統應視為抱怨而非拒絕，回覆說明方案，不改變 proposal state

- 使用者說「不要這個方案」或按「先不要」：系統應 dismiss 當前 proposal instance，不自動切換方案、不永久關閉 rescue、不建立定時 snooze；若使用者只是說「太硬了」，應視為協商而非拒絕

- Dismissed proposal 不應留在 active proposal inbox，但應保留在使用者可見的 history / audit；LINE 不應重推同一個 proposal instance

- History / audit 應顯示簡短摘要，並可展開 explanation；不應直接顯示 raw trace、sidecar diagnostic、或內部推理字彙

## 旅程 F2：預期大餐救援（提前減量）



### 情境



使用者預期週六要吃火鍋吃到飽，希望提前幾天減量來預留空間。



### 前提



旅程 A 已完成（active BodyPlan 存在）



### 測試劇本



1\. 使用者在 chat 說「我這個禮拜六要去吃火鍋吃到飽，我想提前減量」

2\. 系統識別這是預期大餐情境（planned overshoot）

3\. 系統計算：若週六預計多吃約 800 kcal，建議週三到週五每天少吃約 267 kcal

4\. 系統提出預期救援方案，包含：

&#x20;  - 預計超標量（使用者估計或系統預設）

&#x20;  - 建議提前減量的天數與每日調整量

5\. 使用者確認方案

6\. 系統建立 rescue overlay，調整週三到週五的每日目標



### 驗證點



**對話回覆驗證：**

- 系統應識別「提前減量」的意圖，不應只回應「超標了怎麼辦」的被動救援

- 應包含：預計超標量（若使用者未說，可詢問或採預設）、建議提前天數、每日調整量

- 確認後應說明接下來幾天的每日目標



**UI 驗證：**

- 接受後，未來幾天的每日目標應反映 rescue overlay

- Proposal inbox 應顯示預期大餐救援方案



**邊界情境：**

- 使用者未說預計吃多少：系統可詢問，或採保守預設（例如 600 kcal 超標）

## 旅程 G：體重更新（chat 輸入，UI 同步）



### 情境



使用者在 chat 中更新體重，系統自動用新體重重算 TDEE，UI 同步更新。



### 前提



旅程 A 已完成



### 測試劇本



1\. 使用者在 chat 說「我今天量體重是 61 公斤」（原本 onboarding 填 63 公斤）

2\. 系統記錄 BodyObservation（61 kg）

3\. 系統自動用新體重重算 TDEE 與每日目標（不需要問使用者是否要重算）

4\. 系統在 chat 回覆：確認體重已更新，說明新的 TDEE 與每日目標



### 驗證點



**對話回覆驗證：**

- 系統應確認體重已記錄（例如「已記錄體重 61 kg」）

- 系統應說明新的 TDEE 與每日目標（例如「根據新體重，TDEE 約 1750 kcal，每日目標調整為 1400 kcal」）

- 系統**不應**問「是否要重新計算目標？」——體重更新後自動重算，不需確認



**UI 驗證：**

- body-plan 頁面：current\_weight\_kg 更新為 61 kg

- Today 頁面：每日目標反映新計算值

- 體重趨勢圖（若有）：出現新的觀測點



**一致性驗證：**

- chat 回覆的新目標熱量 = UI 顯示的新目標熱量



---



## 旅程 H：體重更新（UI 輸入，chat 可讀）



### 情境



使用者在 UI 的 body-plan 頁面更新體重，之後在 chat 詢問，系統應讀到最新體重。



### 前提



旅程 A 已完成



### 測試劇本



1\. 使用者在 UI body-plan 頁面將體重從 63 kg 改為 61 kg

2\. 系統自動重算 TDEE 與每日目標

3\. 使用者在 chat 問「我現在幾公斤？」或「我今天的目標是多少？」

4\. 系統應回覆最新體重（61 kg）與最新每日目標



### 驗證點



**對話回覆驗證：**

- 系統回覆應使用 UI 更新後的體重（61 kg），不應還是舊的 63 kg

- 系統回覆的每日目標應反映新體重計算結果



**UI 驗證：**

- UI 更新後，body-plan 頁面顯示 61 kg

- Today 頁面每日目標已更新



**一致性驗證：**

- UI 顯示的體重 = chat 回覆的體重

- UI 顯示的每日目標 = chat 回覆的每日目標

## 旅程 I：校準提案（體重趨勢觸發）



### 情境



使用者持續記錄 14 天以上，體重趨勢與預期不符，系統提出校準提案。



### 前提



旅程 A 已完成，旅程 G 或 H 已發生多次（至少 5 次有效體重觀測，14 天視窗）



### 測試劇本



1\. 使用者持續記錄 14 天，每日攝取約 1450 kcal，但體重幾乎沒有下降

2\. 系統判斷 calibration\_confidence 達到 `calibration\\\_candidate`

3\. 系統在 chat 提出校準提案（例如「根據過去兩週的記錄，你的實際消耗可能比預估低，建議將每日目標調整為 1300 kcal」）

4\. 使用者確認提案

5\. 系統更新 active BodyPlan，調整每日目標



### 驗證點



**對話回覆驗證：**

- 提案應說明：觀察到的趨勢、推測原因、建議調整量

- 確認後應回覆新的每日目標



**UI 驗證：**

- 接受後，Today 頁面每日目標更新

- body-plan 頁面顯示新的 calibration 狀態



**邊界情境：**

- 使用者體重記錄不足 5 次：系統不應提出校準提案

- 使用者 intake logging 覆蓋率低於 80%：系統應先提示補記，不提出校準提案



---



## 旅程 J：跳過 Onboarding 的降級行為



### 情境



使用者跳過 onboarding，直接開始使用，系統應降級運作並在適當時機引導。



### 前提



無 active BodyPlan



### 測試劇本



1\. 使用者跳過 onboarding，直接在 chat 說「我剛吃了一碗牛肉麵」

2\. 系統仍可記錄餐點（intake logging 可用）

3\. 系統回覆估算熱量，但**不顯示剩餘預算**（因為沒有 BodyPlan）

4\. 使用者問「我今天還能吃多少？」

5\. 系統回覆：尚未設定目標，引導進入 onboarding



### 驗證點



**對話回覆驗證：**

- 錄入回覆：應給出估算熱量，但不顯示「還剩 X kcal」

- 剩餘預算查詢：應說明尚未設定目標，並提供 onboarding 入口

- 不應說「無法記錄」——intake 仍可用



**UI 驗證：**

- Today 頁面：顯示「尚未設定目標」提示

- 不顯示目標熱量或剩餘熱量

## 旅程 K：餐點修正（UI 數字調整 + chat 自然語言，item-level）



### 情境



使用者記錄了一餐（例如牛肉麵 + 豆漿），事後想修正其中一個或多個 item 的熱量或內容。修正可以透過 UI 直接調整數字，也可以在 chat 用自然語言說。兩種路徑都應正確更新同一個 MealThread 下的對應 MealItem，不影響同餐其他 item。



### 前提



旅程 B 已完成（至少有一筆含多個 item 的 MealThread 存在）



### 測試劇本



**路徑 1：UI 數字調整**



1\. 使用者在 Today UI 的餐點列表中點開「牛肉麵 + 豆漿」這一餐

2\. 點選「豆漿」這個 item

3\. 直接修改熱量數字（例如從 80 kcal 改為 120 kcal），或上下調整

4\. 確認儲存



**路徑 2：chat 自然語言修改**



1\. 使用者在 chat 說「剛才那杯豆漿我記錯了，應該是有糖豆漿，大概 150 kcal」

2\. 系統識別這是對現有 MealThread 中特定 item 的修正

3\. 系統建立新 MealVersion，只更新豆漿的熱量，牛肉麵不變

4\. 系統在 chat 回覆確認修正結果



### 驗證點



**對話回覆驗證（路徑 2）：**

- 系統應確認修正了哪個 item（例如「已將豆漿更新為 150 kcal」）

- 系統應說明這一餐的新總計（例如「這餐現在共約 780 kcal」）

- 牛肉麵的數字不應改變



**UI 驗證（兩條路徑）：**

- Today 頁面：已消耗熱量更新，反映修正後的數字

- 餐點列表：該餐的總計更新，豆漿顯示新數字，牛肉麵不變

- 修正後應建立新 MealVersion（舊版本保留，不覆寫）



**一致性驗證：**

- UI 路徑修正後，chat 問「我今天吃了多少」應讀到修正後的數字

- chat 路徑修正後，UI 應立即反映新數字



### 邊界情境



- 使用者說「把剛才那餐全部刪掉」：系統應確認是否要移除整個 MealThread，不是只改一個 item

- 使用者說「豆漿我沒喝」：系統應將豆漿 item 標記為 `is\\\_removed`，不刪除 MealThread

## 旅程 L：食物推薦（chat 詢問 + UI 瀏覽，點選記錄）



### 情境



使用者想知道現在可以吃什麼。系統根據剩餘熱量、使用者偏好、以及附近可取得的選項，給出推薦。使用者可以在 chat 詢問，也可以在 UI 瀏覽候選，並直接點選記錄。



### 前提



旅程 A 已完成（active BodyPlan 存在，有剩餘預算）



### 測試劇本



**路徑 1：chat 詢問**



1\. 使用者在 chat 說「我現在想吃午餐，有什麼推薦？」

2\. 系統讀取：剩餘熱量（例如 700 kcal）、使用者偏好（例如常吃飯類、偏好中式）、位置（若可用）

3\. 系統在 chat 回覆 1 個主推 + 1–2 個備選（例如「附近的便當店有排骨便當約 650 kcal，符合你的預算，也有雞腿便當約 600 kcal」）

4\. 系統提供 quick actions：`換一個` / `看低熱量` / `幫我記這個`

5\. 使用者點擊「幫我記這個」

6\. 系統轉入 intake flow，記錄該餐點



**路徑 2：UI 瀏覽**



1\. 使用者在 UI 推薦頁面瀏覽候選

2\. UI 顯示多個候選，可依類別篩選（中式 / 低熱量 / 高蛋白）

3\. 使用者點選某個候選，點擊「記錄這餐」

4\. 系統轉入 intake flow，記錄該餐點



### 驗證點



**對話回覆驗證（路徑 1）：**

- 推薦應包含：食物名稱、估算熱量、為什麼推薦（例如「符合你的剩餘預算，也是你常吃的類型」）

- 推薦的熱量應在剩餘預算範圍內

- 提供 quick actions，包含 `幫我記這個`

- 點擊 `幫我記這個` 後，系統應確認記錄，Today 頁面更新



**UI 驗證（路徑 2）：**

- 候選列表顯示估算熱量範圍

- 篩選器可用（類別、熱量範圍）

- 點選記錄後，Today 頁面更新，chat 可讀到新記錄



**一致性驗證：**

- chat 推薦的候選 = UI 推薦頁面的候選（同一批 ranking 結果）

- 記錄後，chat 問「我今天吃了什麼」應包含剛記錄的推薦餐點



### 邊界情境



- 剩餘預算不足 300 kcal：推薦應只顯示低熱量選項，不推薦超出預算的餐點

- 位置不可用：系統應退回歷史偏好 + 常見選項，不報錯

- 使用者說「換一個」：系統給出不同候選，不重複剛才的主推

## 旅程 M：偏好記憶（明確聲明 + 行為歸納，推薦時展現）



### 情境



系統應能記住使用者的飲食偏好，並在推薦時展現出這份理解。偏好來源有兩種：使用者明確聲明，以及系統從歷史餐點歸納。



### 前提



旅程 L 已完成（推薦功能就位）；路徑 2 需要至少 21 筆 committed MealItem 歷史



### 測試劇本



**路徑 1：明確聲明**



1\. 使用者在 chat 說「記住我不喝含糖飲料」

2\. 系統確認已記住（例如「好的，之後推薦飲料時會優先選無糖選項」）

3\. 使用者之後詢問推薦

4\. 系統推薦的飲料選項應全為無糖，不出現含糖飲料



**路徑 2：行為歸納**



1\. 使用者累積記錄 21 筆以上餐點，其中大多數午餐是飯類（便當、滷肉飯等）

2\. 系統在背景完成 semantic pattern extraction

3\. 使用者詢問午餐推薦

4\. 系統推薦應以飯類為主推，麵類為備選，而不是隨機排序



### 驗證點



**對話回覆驗證（路徑 1）：**

- 聲明後，系統應明確確認已記住

- 之後的推薦回覆中，飲料選項不應出現含糖飲料

- 若使用者問「你記得我不喝什麼嗎？」，系統應能回答



**對話回覆驗證（路徑 2）：**

- 午餐推薦的主推應是飯類

- 推薦說明中應體現偏好理解（例如「根據你的習慣，推薦飯類選項」）



**邊界情境：**

- 使用者說「我最近開始不吃油炸食物了」：系統應更新偏好，之後推薦不出現油炸選項

- 使用者說「今天想試試麵食」：當下聲明優先於歷史統計，這次推薦給麵食，但不改變長期偏好

## 旅程 N：Proactive 主動介入（提醒、推薦、rescue nudge）



> \\\*\\\*注意：此旅程依賴 Slice 2.9 Proactive Nudges，是最後就位的能力。現在記錄的是產品理想狀態的骨架，供 2.9 實作時對齊。\\\*\\\*



### 情境



系統在使用者沒有主動詢問的情況下，主動發出有用的訊息。這是產品最核心的 agentic 能力——不只是記錄工具，而是真正幫助使用者達成目標的智能助手。



### 前提



旅程 A–M 的對應能力就位；2.9 Proactive Nudges slice 完成



### 子情境 N-1：進餐提醒（尚未記錄）



**劇本：**

1\. 使用者平常午餐約 12:30 記錄，今天 13:00 還沒有任何記錄

2\. 系統主動發送提醒（例如「你今天午餐還沒記錄，要幫你記嗎？」）

3\. 使用者回覆「我剛吃了雞腿便當」

4\. 系統直接進入 intake flow



**驗證點：**

- 提醒應是獨立訊息，不夾帶在其他回覆裡

- 提醒時機應符合使用者的歷史用餐時段，不應在不合理時間發送

- 使用者可以說「先不管」，系統不再重複提醒（suppression 生效）



### 子情境 N-2：量體重提醒



**劇本：**

1\. 使用者已超過 3 天沒有記錄體重

2\. 系統主動提醒（例如「你已經 3 天沒量體重了，記錄一下有助於追蹤進度」）

3\. 使用者回覆「我今天 62 公斤」

4\. 系統記錄 BodyObservation，自動重算 TDEE



**驗證點：**

- 提醒頻率應受 cooldown 控制，不應每天都提醒

- 使用者可以關閉體重提醒，系統不再發送



### 子情境 N-3：Rescue Nudge（超標後主動提醒）



**劇本：**

1\. 使用者今日超標 500 kcal，但沒有主動詢問救援

2\. 系統在適當時機（非用餐時段）主動發送 rescue 提案

3\. 使用者可接受、用 chat 協商調整，或先不要



**驗證點：**

- Rescue nudge 應是獨立訊息

- 不應在使用者剛記錄完餐點時立即發送（intake reply 與 rescue 分離）

- 使用者按「先不要」後，應只 dismiss 當前 proposal instance；若後續情境重大改變或使用者主動重開討論，系統才可生成新的 proposal

- Dismissed proposal 應離開 active inbox，但仍可在 history / audit 中被使用者看到

- History / audit 應提供 concise summary + expandable explanation，不直接暴露 raw trace 或 sidecar diagnostic vocabulary

- 使用者忽略後，系統不應在同一天重複發送



### 子情境 N-4：主動食物推薦（符合目標的選項）



**劇本：**

1\. 使用者今日剩餘 600 kcal，接近晚餐時段（18:00）

2\. 系統根據使用者偏好與剩餘預算，主動推薦晚餐選項（例如「你今晚還有 600 kcal，附近的 XX 便當店有雞腿便當約 550 kcal，符合你的習慣」）

3\. 使用者可直接點選記錄，或說「換一個」



**驗證點：**

- 推薦應符合剩餘預算（不超標）

- 推薦應體現偏好理解（不是隨機選項）

- 使用者可以說「今天不想被推薦」，系統當日不再主動推薦

- 推薦時機應在合理用餐時段，不應在深夜發送



### 共同驗證原則



- 所有 proactive 訊息都是獨立訊息，不夾帶在其他回覆裡

- 所有 proactive 訊息都受 cooldown / suppression 控制

- 使用者可以在 chat 說「不要再提醒我 XX」，系統應記住並停止該類提醒

- quiet hours 設定應被尊重（例如 22:00–08:00 不發送任何 proactive 訊息）

## 旅程 O：照片錄入（chat 拍照 / 上傳，UI 顯示附圖）



> \\\*\\\*注意：此旅程依賴 multimodal 輸入能力，屬於後期功能。現在記錄骨架供對齊。\\\*\\\*



### 情境



使用者在 chat 對話框拍照或上傳食物照片，系統識別食物內容並估算熱量，走標準 intake flow。UI 的餐點列表中，該餐點會附上對應的照片。



### 設計邊界



- **照片輸入**：僅限 chat 對話框，UI 不提供照片上傳入口

- **照片顯示**：UI 餐點列表中，有照片的餐點會顯示縮圖，照片與 MealThread 綁定

- **語音輸入**：不在本旅程，見旅程 P



### 前提



旅程 A 已完成；multimodal 輸入能力就位



### 測試劇本



1\. 使用者在 chat 對話框上傳一張食物照片（例如一碗牛肉麵的照片）

2\. 系統識別照片內容，回覆估算結果（例如「看起來是牛肉麵，約 600–750 kcal，已記錄為約 680 kcal」）

3\. 若照片內容不清楚，系統應反問（例如「這是什麼料理？可以描述一下嗎？」）

4\. 記錄完成後，UI 餐點列表中該餐點顯示照片縮圖



### 驗證點



**對話回覆驗證：**

- 系統應識別照片中的食物，給出估算範圍與採用值

- 若照片模糊或無法識別，應反問，不應直接說「無法估算」後停止

- 估算結果應合理（不應給出明顯偏低或偏高的數字）



**UI 驗證：**

- Today 頁面餐點列表：該餐點顯示照片縮圖

- 照片與 MealThread 正確綁定（不會出現在其他餐點旁邊）

- 消耗熱量正確更新



**邊界情境：**

- 照片包含多種食物（例如便當盒）：系統應嘗試逐項識別，或給出整體估算

- 使用者上傳非食物照片：系統應說明無法識別為食物，引導重新輸入

**邊界情境：**

## 旅程 Q：餐前情境規劃（「等一下去哪吃」）

### 情境

使用者說「等一下要去信義區吃飯」，系統根據剩餘預算、偏好、位置，提前給出建議。

### 前提

旅程 A 已完成；旅程 L 就位

### 測試劇本

1. 使用者在 chat 說「我等一下要去信義區吃飯，有什麼推薦？」
2. 系統讀取剩餘預算、偏好、位置（信義區）
3. 系統走 L3.2 pre-meal planning mode，給出附近符合偏好的建議
4. 使用者說「就去這家」，系統說明預算分配建議

### 驗證點

**對話回覆驗證：**
- 系統應識別「等一下去哪裡」的 pre-meal planning intent
- 推薦應包含地點相關資訊（附近店家）
- 應說明這餐建議的熱量範圍，讓使用者知道還剩多少空間

**邊界情境：**
- 位置不可用：退回歷史偏好推薦，不報錯

## 旅程 R：外食菜單掃描（拍菜單 → 推薦點什麼）

### 情境

使用者在餐廳拍下菜單，系統根據剩餘預算推薦點什麼。這是決策前的推薦，不是記錄已吃的。

### 前提

旅程 A 已完成；multimodal 能力就位

### 測試劇本

1. 使用者在 chat 上傳菜單照片，說「我在這家餐廳，有什麼可以點？」
2. 系統解析菜單（vision_parser_model），提取菜單項目與估算熱量
3. 系統根據剩餘預算 + 偏好，推薦 1–2 個選項
4. 使用者說「就點這個」，系統轉入 intake flow 記錄

### 驗證點

**對話回覆驗證：**
- 系統應識別這是「決策前」的菜單掃描，不是記錄已吃的
- 推薦應包含：菜名、估算熱量、為什麼推薦
- 推薦的熱量應在剩餘預算範圍內
- 使用者說「就點這個」後，系統轉入 intake flow，Today 頁面更新

**邊界情境：**
- 菜單照片模糊：系統應說明無法解析，引導使用者手動描述
- 菜單全部超出預算：系統應說明並建議最接近預算的選項

## 旅程 S：食物替換建議（Swap Suggestion）

### 情境

使用者記錄了高熱量食物後，系統主動提出具體替換建議（「下次可以換成什麼」）。

### 前提

旅程 B 已完成；使用者有足夠歷史記錄（非 cold-start）

### 測試劇本

1. 使用者記錄「全糖大杯珍珠奶茶」（約 700 kcal）
2. 系統完成 intake 記錄後，作為獨立訊息提出替換建議
3. 例如：「全糖大杯珍奶約 700 kcal，如果改成半糖每次省約 150 kcal，一週省約 1050 kcal」
4. 使用者說「記住這個建議」，系統寫入 confirmed preference memory

### 驗證點

**對話回覆驗證：**
- swap suggestion 應是獨立訊息，不夾帶在 intake reply 裡
- 應包含：原食物熱量、替換選項、每次省多少、一週省多少
- 建議應具體可執行，不是泛泛的「少吃一點」

**邊界情境：**
- Cold-start 使用者（歷史不足）：不觸發 swap suggestion
- 使用者說「不用了」：系統不再推送這類建議（suppression）

## 旅程 T：聚餐情境（全天熱量分配）

### 情境

使用者說「今晚有聚餐，不知道會吃多少」，系統幫助規劃全天熱量分配。

### 前提

旅程 A 已完成

### 測試劇本

1. 使用者在 chat 說「今晚有公司聚餐，我想提前規劃一下」
2. 系統識別聚餐情境，計算今日剩餘預算
3. 系統建議白天保留空間（例如「建議今晚保留 600 kcal，午餐控制在 400 kcal 以內」）
4. 使用者說「幫我設定今晚保留 800 kcal」
5. 系統升格為 planned_event_budget_allocation proposal，使用者確認後建立 rescue overlay

### 驗證點

**對話回覆驗證：**
- 系統應識別聚餐情境，給出全天分配建議
- 若使用者只是詢問建議：informational 回覆，不建立 proposal
- 若使用者明確說「幫我設定」：升格為 proposal，走 L3.4 Section 17A

**UI 驗證：**
- 確認 proposal 後，未來幾天的每日目標應反映 rescue overlay

## 旅程 U：運動量輸入（今日 TDEE 臨時調整）

### 情境

使用者記錄今日運動量，系統計算消耗熱量並調整今日可進食預算。

### 前提

旅程 A 已完成

### 測試劇本

1. 使用者在 chat 說「我今天跑步 30 分鐘」
2. 系統識別運動類型（running）和時長（30 分鐘）
3. 系統用 MET formula 計算消耗（例如 244 kcal）
4. 系統建立 ExerciseEvent + LedgerEntry(exercise_bonus)
5. 系統回覆：「跑步 30 分鐘消耗約 244 kcal，今天可以多吃 244 kcal，有效預算調整為 1644 kcal」

### 驗證點

**對話回覆驗證：**
- 系統應確認運動類型和消耗熱量
- 系統應說明今日新的有效預算
- 系統**不應**問「是否要調整目標？」——直接調整，不需確認
- 系統**不應**改寫 BodyPlan.estimated_tdee——只影響今日 ledger

**UI 驗證：**
- Today 頁面：有效預算更新，反映 exercise_bonus
- 消耗熱量不變，但可進食上限增加

**一致性驗證：**
- chat 回覆的新有效預算 = UI 顯示的新有效預算

**邊界情境：**
- 使用者說「我今天消耗了 300 kcal」：採 user_asserted，直接使用 300 kcal，不重算
- 無法識別運動類型：系統反問，不猜測

## 旅程 V：週報洞察（定期行為 insight）

> **注意：此旅程依賴 Slice 2.9 Proactive Nudges。現在記錄骨架。**

### 情境

系統每週主動發送一份行為洞察報告，幫助使用者理解自己的飲食模式。

### 前提

旅程 A 已完成；使用者有至少 7 天使用歷史；本週 intake logging coverage ≥ 50%

### 測試劇本

1. 每週一 08:00，系統生成 WeeklyInsightReport
2. 系統作為獨立 chat 訊息發送週報
3. 例如：「你這週有 4 天達標，超標都集中在週三晚上。飲料佔了本週熱量的 18%，如果改成無糖每週可以多出約 700 kcal 的空間。」

### 驗證點

**對話回覆驗證：**
- 週報應是獨立訊息
- 應包含：達標率、超標模式、熱量來源分析、具體可執行建議
- 數字應來自 deterministic 計算，narrative 由 LLM 生成
- 不應是模板訊息，應體現個人化

**邊界情境：**
- 本週 logging coverage < 50%：不生成週報，或生成降級版本（只說「這週記錄不足，無法分析」）
